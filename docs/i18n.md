# Locales and Translations

## Locale model

`CmsLocale` includes:

- `code` (e.g. `en`, `nl`, `nl-BE`)
- `language`
- optional `country`
- optional `default` (resolved from channel when available)

Helpers normalize locale casing (`nl-be` -> `nl-BE`) and split language/country safely.

## Translated routes

`CmsRoute.translations` stores locale-to-path mapping.

Example:

```json
{
  "en": "/contact-us",
  "nl": "/contacteer-ons"
}
```

Resolver can match incoming slugs against any translation and determine canonical output for the selected locale.

## SEO alternates

Metadata helper can emit alternates:

- canonical URL
- language URLs (`hreflang` style mapping)

Use `localeToHrefLang` when your locale code and hreflang format differ.

When `routing` is provided to `buildNextMetadataFromPage`:

- `language` strategy emits language-only alternates (`en`, `nl`)
- `country-language` strategy emits country-language alternates (`en-BE`, `nl-BE`)
- you can force full language x country expansion using `alternateLanguages` + `alternateCountries`

## Localized slugs for static routes

For hard-coded pages that are not CMS routes, use locale-to-slug maps (JSON-friendly) with Next helpers:

```ts
import { buildLocalizedSlugAlternates, buildLocalizedStaticPath } from "@ominity/next/next";

const slugByLocale = {
  en: "contact",
  nl: "contacteer-ons",
};

const canonicalPath = buildLocalizedStaticPath({
  routing,
  locale: "nl-BE",
  slugByLocale,
});

const { alternates } = buildLocalizedSlugAlternates({
  routing,
  locale: "nl-BE",
  slugByLocale,
  baseUrl: "https://www.example.com",
  // optional cartesian expansion (country-language strategy):
  countries: ["BE", "NL"],
  languages: ["en", "nl"],
});
```

Resolution order is:

1. exact locale (`nl-BE`)
2. language fallback (`nl`)
3. default locale fallback

Configured routing/channel locales remain the source of truth for which locales are emitted in alternates.

## Locale-aware API requests

CMS client supports locale-aware requests:

- locale query parameter
- `Accept-Language` header
- `X-Channel-Id` header

so data fetch behavior is explicit and environment-independent.

## Channel-aware locale resolution

With `@ominity/api-typescript@^1.0.0`, locales are read from `/localization/languages`.

When `/channels/current` is available, `getLocales()` also merges channel configuration:

- available channel languages
- available channel countries
- default language/country

This means routing/localization can be driven by channel settings instead of hardcoded locale arrays.

## Practical behavior

- **No channel endpoint**: locales come from `/localization/languages`.
- **Channel endpoint available**: locales are filtered/augmented by channel language + country configuration.
- **Default locale**: resolved from channel default language/country when present.
